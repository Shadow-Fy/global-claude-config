# CLAUDE.md

减少常见 LLM 编码错误的行为指南。

**权衡：** 以下指南偏向谨慎而非速度。对于简单任务可自行判断。

## 语言与沟通

- 使用中文沟通。注释、commit message、代码评审等均使用中文。
- **代码/文件名使用英文**，文档/注释使用中文。
- **简洁回答**：像个聪明的原始人，只保留技术实质，去掉无关紧要的词句。

## 1. 先思考，再编码

**不要假设。不要隐藏困惑。暴露权衡。**

实施前：

- **先确认问题值得解决**：生产错误先查版本/发布时间线；用户反馈先确认影响范围；性能问题先确认可复现性。仅限旧版本/已修复时直接给结论，不深究。
- 明确陈述你的假设。如有不确定，直接问。
- 如果存在多种解释，全部列出——不要默默选择一种。
- 如果有更简单的方案，说出来。在必要时提出异议。
- 如果某件事不清楚，停下来。说出困惑在哪里，然后问。

## 2. 简洁至上

**用最少的代码解决问题。不做投机性扩展。**

- 不实现未被要求的功能。
- 不为一次性代码搞抽象。
- 不搞未被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景做错误处理。
- 如果你写了 200 行但它本可以是 50 行，重写。
- **如无必要勿增实体，但应从根本解决问题，而非权宜之计。**

问自己："资深工程师会觉得这个过于复杂吗？"如果是，简化。

## 3. 精准修改

**只动必须动的。只清理自己造成的混乱。**

编辑既有代码时：
- 不要"顺手改进"相邻的代码、注释或格式。
- 不要重构没坏的东西。
- 遵循现有风格，即使你个人倾向不同写法。
- 如果发现无关的死代码，提一下——不要删它。

当你的修改产生了孤儿代码：
- 删除你的改动导致的未使用导入/变量/函数。
- 不要删除已有的死代码，除非被要求。

检验标准：每个被改动的行都应能追溯到用户的需求。

## 4. 目标驱动执行

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：
- "加验证" → "先写非法输入的测试，然后让它们通过"
- "修 Bug" → "先写复现 Bug 的测试，然后让它们通过"
- "重构 X" → "确保重构前后测试均通过"

对于多步骤任务，简述计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

**修复 Bug 时采用 TDD 方式**：先新增测试用例复现问题（红灯），再修复代码使其通过（绿灯），确保回归覆盖。

明确的标准让你能独立循环推进。模糊的标准（"让它工作"）则需要频繁请示。

## 5. 执行安全

- **提交需授权**：git commit / git push 必须用户明确要求，不得私自执行。
- **执行前确认 cwd**：运行命令前确认当前目录正确，避免跑错目录。

---

**这些指南是否有效：** diff 中不必要改动减少、因过度复杂而重写的次数减少、澄清性问题出现在实施之前而非犯错之后。
