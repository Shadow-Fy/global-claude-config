# ADR 格式

ADR 位于 `docs/adr/` 目录，使用顺序编号：`0001-slug.md`、`0002-slug.md` 等。

懒创建 `docs/adr/` 目录——只在需要第一个 ADR 时才创建。

## 模板

```md
# {决策的简短标题}

{1-3 句话：上下文是什么，我们做了什么决定，以及为什么。}
```

就是这样。一个 ADR 可以只是一个段落。其价值在于记录**做出了**一个决策以及**为什么**——而不是填充各个章节。

## 可选章节

仅在能带来真正价值时才包含这些。大多数 ADR 不需要它们。

- **Status** 元数据（`proposed | accepted | deprecated | superseded by ADR-NNNN`）——当决策被重新审视时有用
- **Considered Options**（考虑过的选项）——仅当被拒绝的替代方案值得记住时
- **Consequences**（后果）——仅当有非显而易见的后续影响需要指出时

## 编号

扫描 `docs/adr/` 找到现有最高编号，加一。

## 何时提供 ADR

以下三个条件必须全部满足：

1. **难以逆转**——事后改变主意的成本很高
2. **没有上下文会令人惊讶**——未来的读者看到代码会想"他们到底为什么这么做？"
3. **真实权衡的结果**——存在真正的替代方案，你因特定原因选择了其中一个

如果一个决策容易逆转，跳过它——你以后会撤销它。如果不令人惊讶，没人会好奇为什么。如果没有真正的替代方案，除了"我们做了显而易见的事情"之外没有什么可记录的。

### 什么算作合格的 ADR

- **架构形状**。"我们在用单体仓库。""写模型是事件溯源的，读模型投影到 Postgres。"
- **上下文之间的集成模式**。"Ordering 和 Billing 通过领域事件通信，而不是同步 HTTP。"
- **带有锁定效应的技术选择**。数据库、消息总线、认证提供商、部署目标。不是每个库——只是那些换掉需要一个季度的。
- **边界和范围决策**。"客户数据归 Customer 上下文所有；其他上下文仅通过 ID 引用。"明确说不和说是同样有价值。
- **对显而易见路径的故意偏离**。"我们使用手动 SQL 而不是 ORM，因为 X。"任何合理读者会假设相反情况的东西。这可以阻止下一位工程师"修复"本属有意为之的东西。
- **代码中不可见的约束**。"由于合规要求，我们不能使用 AWS。""由于合作伙伴 API 合同，响应时间必须低于 200ms。"
- **被拒绝的替代方案（当拒绝原因非显而易见时）**。如果你考虑了 GraphQL 而因微妙的理由选择了 REST，记录下来——否则六个月后又会有人建议 GraphQL。
