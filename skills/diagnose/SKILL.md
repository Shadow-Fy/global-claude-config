# 诊断

针对棘手 Bug 的一套规范流程。只有在有明确理由时才跳过某个阶段。

在探索代码库时，使用项目的领域术语表来建立相关模块的清晰心智模型，并查看你涉及的区域的 ADR。

## 第一阶段 — 构建反馈循环

**这是核心技能。** 其他都是机械性的。如果你有一个快速、确定性的、可由 agent 运行的 Bug 通过/失败信号，你就能找到原因——二分查找、假设检验和检测都只是消费这个信号而已。如果没有这个信号，盯着代码看再多也没用。

在此投入不成比例的努力。**要有攻击性。要有创造力。拒绝放弃。**

### 构建反馈循环的方法——大致按此顺序尝试

1. **失败测试**——在能触及 Bug 的任何接缝处编写：单元、集成、端到端。
2. **Curl / HTTP 脚本**——对着运行中的开发服务器执行。
3. **CLI 调用**——使用一个固定的输入文件，将 stdout 与已知良好的快照进行 diff。
4. **无头浏览器脚本**（Playwright / Puppeteer）——驱动 UI，在 DOM/控制台/网络上断言。
5. **重放捕获的跟踪**——将真实的网络请求/负载/事件日志保存到文件；通过代码路径隔离地重放。
6. **一次性测试工具**——启动系统的最小子集（一个服务，mock 依赖），通过单个函数调用执行有 Bug 的代码路径。
7. **属性/模糊测试循环**——如果 Bug 是"偶尔输出错误"，运行 1000 个随机输入，观察失败模式。
8. **二分查找工具**——如果 Bug 出现在两个已知状态（提交、数据集、版本）之间，自动化"在状态 X 启动、检查、重复"以便 `git bisect run`。
9. **差分循环**——将相同输入通过旧版本 vs 新版本（或两种配置）运行，diff 输出。
10. **人工介入脚本**——最后手段。如果需要人工点击，用 `scripts/hitl-loop.template.sh` 驱动**他们**，使循环仍然结构化。捕获的输出反馈给你。

构建正确的反馈循环，Bug 就已经修复了 90%。

### 在循环本身上迭代

把循环当作产品来对待。一旦你**有了**一个循环，问问：

- 能不能更快？（缓存设置，跳过无关初始化，缩小测试范围。）
- 能不能让信号更清晰？（断言具体的症状，而不是"没有崩溃"。）
- 能不能让它更确定？（固定时间、随机种子、隔离文件系统、冻结网络。）

一个 30 秒的不稳定循环比没有循环好不了多少。一个 2 秒的确定性循环是调试的超能力。

### 非确定性 Bug

目标不是完美的复现，而是**更高的复现率**。将触发循环 100 次，并行化，增加压力，缩小时间窗口，注入 sleep。50% 概率的 Bug 是可以调试的；1% 则不行——持续提高复现率直到可以调试。

### 当你确实无法构建循环时

停下来，明确说出来。列出你尝试过的方法。向用户请求：(a) 访问能复现该 Bug 的环境，(b) 捕获的产物（HAR 文件、日志转储、核心转储、带时间戳的屏幕录像），或 (c) 添加临时生产检测的权限。**不要**在没有循环的情况下继续假设。

在拥有一个你信赖的循环之前，不要进入第二阶段。

## 第二阶段 — 复现

运行循环。观察 Bug 出现。

确认：

- [ ] 循环产生了**用户**描述的失败模式——而不是恰好附近的其他失败。错误的 Bug = 错误的修复。
- [ ] 失败在多次运行中可复现（或对于非确定性 Bug，在足够高的概率下可调试）。
- [ ] 你已经捕获了确切的症状（错误消息、错误输出、慢速），以便后续阶段验证修复确实解决了问题。

在复现 Bug 之前不要继续。

## 第三阶段 — 假设

在测试任何一个假设之前，生成 **3-5 个排序的假设**。单一假设生成会让你锚定在第一个看似合理的想法上。

每个假设必须是**可证伪的**：陈述它做出的预测。

> 格式："如果 <X> 是原因，那么 <改变 Y> 将使 Bug 消失 / <改变 Z> 将使情况更糟。"

如果你无法陈述预测，这个假设只是一种感觉——丢弃它或精炼它。

**在测试之前将排序列表展示给用户。** 他们通常拥有能立即重新排序的领域知识（"我们刚部署了对 #3 的修改"），或者知道他们已经排除的假设。廉价检查点，省时省力。如果用户不在，按你的排序继续。

## 第四阶段 — 检测

每次检测必须对应第三阶段的一个具体预测。**一次只改变一个变量。**

工具偏好：

1. **调试器 / REPL 检查**——如果环境支持。一个断点胜过十个日志。
2. **定向日志**——在能区分假设的边界上打印。
3. 永远不要"全部记录然后 grep"。

**为每个调试日志打上唯一前缀标签**，例如 `[DEBUG-a4f2]`。最后清理时只需一次 grep。无标签的日志保留；有标签的删除。

**性能分支**。对于性能回退，日志通常是错的。改为：建立基线测量（计时工具、`performance.now()`、分析器、查询计划），然后二分。先测量，再修复。

## 第五阶段 — 修复 + 回归测试

**在修复之前**编写回归测试——但前提是存在**正确的接缝**。

正确的接缝是指测试能在调用点处实际发生的**真实 Bug 模式**上进行测试。如果唯一可用的接缝太浅（当 Bug 需要多个调用者时只有一个单调用者的测试，无法复现触发 Bug 的调用链的单元测试），在那里放回归测试会产生虚假的信心。

**如果没有正确的接缝，这本身就是一个发现。** 记录下来。代码库架构阻止了这个 Bug 被锁定。将其标记为下一阶段的问题。

如果存在正确的接缝：

1. 将最小化的复现转化为该接缝处的失败测试。
2. 观察它失败。
3. 应用修复。
4. 观察它通过。
5. 针对原始（未最小化）场景重新运行第一阶段的反馈循环。

## 第六阶段 — 清理 + 事后总结

在宣布完成之前必须完成：

- [ ] 原始复现不再复现（重新运行第一阶段的循环）
- [ ] 回归测试通过（或缺少接缝的情况已记录）
- [ ] 所有 `[DEBUG-...]` 检测已移除（`grep` 该前缀）
- [ ] 一次性的测试原型已删除（或移到明确标记的调试位置）
- [ ] 正确的假设已记录在 commit / PR 消息中——以便下一个调试者学习

**然后问：什么本可以防止这个 Bug？** 如果答案涉及架构变更（没有好的测试接缝、调用者纠缠不清、隐藏的耦合），将具体的发现交给 `/improve-codebase-architecture` skill。在修复完成**之后**再做推荐——你现在比开始时拥有更多信息。
