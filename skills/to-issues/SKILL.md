# 转为问题

使用垂直切片（tracer bullets）将计划分解为可独立领取的问题。

问题跟踪器和分类标签词汇表应该已经提供给你——如果没有，运行 `/setup-matt-pocock-skills`。

## 流程

### 1. 收集上下文

从对话上下文中已有的内容开始。如果用户传入了问题引用（问题编号、URL 或路径）作为参数，从问题跟踪器中获取并读取其完整正文和评论。

### 2. 探索代码库（可选）

如果你还没有探索过代码库，先这样做以了解代码的当前状态。问题标题和描述应使用项目的领域术语表词汇，并尊重你正在涉及区域的 ADR。

### 3. 起草垂直切片

将计划分解为 **tracer bullet** 问题。每个问题是一个薄的垂直切片，端到端贯穿所有集成层，**不是**一个层的水平切片。

切片可以是 'HITL'（需人工参与）或 'AFK'（无需人工参与）。HITL 切片需要人机交互，例如架构决策或设计评审。AFK 切片可以在没有人机交互的情况下实现和合并。尽可能优先选择 AFK 而非 HITL。

<vertical-slice-rules>
- 每个切片提供一条狭窄但**完整**的路径贯穿每一层（schema、API、UI、测试）
- 一个完成的切片本身是可演示或可验证的
- 宁多薄切片，勿少厚切片
</vertical-slice-rules>

### 4. 提问用户

将建议的分解呈现为编号列表。对每个切片，展示：

- **标题**：简短的描述性名称
- **类型**：HITL / AFK
- **被谁阻塞**：哪些其他切片（如果有）必须先完成
- **涵盖的用户故事**：这解决了哪些用户故事（如果源材料有的话）

问用户：

- 粒度感觉合适吗？（太粗 / 太细）
- 依赖关系正确吗？
- 是否有切片应该合并或进一步拆分？
- HITL 和 AFK 的标记正确吗？

迭代直到用户批准分解方案。

### 5. 将问题发布到问题跟踪器

对于每个批准的切片，向问题跟踪器发布一个新问题。使用下面的问题正文模板。这些问题被视为已准备好供 AFK agent 使用，因此除非另有指示，请使用正确的分类标签发布。

按照依赖顺序发布问题（先发布阻塞项），以便你可以在"被谁阻塞"字段中引用真实的问题标识符。

<issue-template>
## 父级

对父问题的问题跟踪器引用（如果源是现有问题，则包含此部分；否则省略）。

## 要构建什么

此垂直切片的简洁描述。描述端到端的行为，而不是逐层实现。

避免具体的文件路径或代码片段——它们很快就会过时。例外：如果原型产生了比散文更精确地编码决策的片段（状态机、reducer、schema、类型形状），在此处内联它，并简要说明它来自原型。裁剪到决策丰富的部分——不是工作演示，只是重要的部分。
