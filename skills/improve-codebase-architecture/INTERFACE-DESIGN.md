# 接口设计

当用户想为选定的深化候选探索替代接口时，使用此并行子 agent 模式。基于"Design It Twice"（Ousterhout）——你的第一个想法不太可能是最好的。

使用 [LANGUAGE.md](LANGUAGE.md) 中的词汇表——**module**、**interface**、**seam**、**adapter**、**leverage**。

## 流程

### 1. 框定问题空间

在启动子 agent 之前，为选定的候选编写面向用户的问题空间解释：

- 任何新接口需要满足的约束
- 它依赖的依赖项，以及它们属于哪个类别（见 [DEEPENING.md](DEEPENING.md)）
- 一个粗略的说明性代码草图来具体化约束——不是一个方案，只是让约束具体化的一种方式

向用户展示，然后立即进入第 2 步。用户在子 agent 并行工作时阅读和思考。

### 2. 启动子 agent

使用 Agent 工具并行启动 3 个以上的子 agent。每个必须为深化后的模块产生一个**根本不同**的接口。

为每个子 agent 提供独立的技术简报（文件路径、耦合细节、[DEEPENING.md](DEEPENING.md) 中的依赖分类、接缝后面是什么）。该简报独立于第 1 步中面向用户的问题空间解释。给每个 agent 不同的设计约束：

- Agent 1："最小化接口——最多 1-3 个入口点。最大化每个入口点的杠杆。"
- Agent 2："最大化灵活性——支持多种用例和扩展。"
- Agent 3："针对最常见的调用者优化——让默认情况变得简单。"
- Agent 4（如适用）："围绕端口与适配器设计跨接缝的依赖。"

在简报中同时包含 [LANGUAGE.md](LANGUAGE.md) 词汇表和 CONTEXT.md 词汇表，以便每个子 agent 在命名时与架构语言和项目领域语言一致。

每个子 agent 输出：

1. 接口（类型、方法、参数——以及不变式、顺序、错误模式）
2. 使用示例，展示调用者如何使用
3. 实现隐藏在接缝后面的内容
4. 依赖策略和适配器（见 [DEEPENING.md](DEEPENING.md)）
5. 权衡——杠杆高的地方，杠杆薄弱的地方

### 3. 呈现与比较

按顺序呈现设计，以便用户吸收每个设计，然后用文字进行比较。根据**深度**（接口处的杠杆）、**局部性**（变更集中的地方）和**接缝放置**进行对比。

比较后，给出你自己的推荐：你认为哪个设计最强以及为什么。如果不同设计的元素可以良好结合，提出混合方案。要有主见——用户想要的是强有力的判断，而不是菜单。
