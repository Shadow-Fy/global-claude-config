# 深化

如何安全地深化一组浅层模块，考虑到它们的依赖关系。假定使用 [LANGUAGE.md](LANGUAGE.md) 中的词汇表——**module**、**interface**、**seam**、**adapter**。

## 依赖分类

在评估深化候选时，对其依赖进行分类。分类决定了深化后的模块如何在其接缝处进行测试。

### 1. 进程内

纯计算、内存状态、无 I/O。总是可以深化——合并模块，通过新接口直接测试。不需要适配器。

### 2. 本地可替代

有本地测试替代品的依赖（PGLite 替代 Postgres，内存文件系统）。如果存在替代品则可深化。深化后的模块使用测试套件中的替代品进行测试。接缝是内部的；在模块的外部接口处没有端口。

### 3. 远程但自有（端口与适配器）

你自有的跨网络边界的服务（微服务、内部 API）。在接缝处定义一个 **port**（接口）。深度模块拥有逻辑；传输层作为 **adapter** 注入。测试使用内存适配器。生产环境使用 HTTP/gRPC/队列适配器。

推荐表述：*"在接缝处定义一个 port，为生产环境实现 HTTP 适配器，为测试环境实现内存适配器，这样即使跨网络部署，逻辑也位于一个深度模块中。"*

### 4. 真正的外部依赖（Mock）

你无法控制的第三方服务（Stripe、Twilio 等）。深化后的模块将外部依赖作为注入的 port 接收；测试提供一个 mock 适配器。

## 接缝规范

- **一个适配器意味着假设的接缝。两个适配器意味着真实的接缝。** 不要引入 port，除非至少有两个适配器是合理的（通常是生产环境 + 测试环境）。单适配器的接缝只是间接层。
- **内部接缝 vs 外部接缝。** 深度模块可以有内部接缝（对其实现私有，供自身测试使用）以及其接口处的外部接缝。不要仅仅因为测试使用它们就将内部接缝暴露到接口中。

## 测试策略：替换，而非分层

- 一旦深度模块接口处的测试存在，浅层模块上的旧单元测试就变成浪费——删除它们。
- 在深度模块的接口处编写新测试。**接口就是测试面。**
- 测试通过接口断言可观察的结果，而不是内部状态。
- 测试应该能够经受内部重构——它们描述行为，而非实现。如果测试在实现改变时必须改变，那它就是在测试越过接口。
