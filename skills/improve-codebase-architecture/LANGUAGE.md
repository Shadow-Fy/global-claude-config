# 语言

此技能每条建议共享的词汇表。精确使用这些术语——不要替换为"component"、"service"、"API"或"boundary"。一致的语言是整个要点。

## 术语

**Module（模块）**
任何有接口和实现的东西。有意保持规模无关——同样适用于函数、类、包或跨层切片。
_避免使用_：unit, component, service。

**Interface（接口）**
调用者必须知道才能正确使用该模块的一切。包括类型签名，但也包括不变式、顺序约束、错误模式、所需配置和性能特征。
_避免使用_：API, signature（太窄——它们只指类型层面的表面）。

**Implementation（实现）**
模块内部的东西——它的代码体。与 **Adapter** 区分：一个东西可以是小的适配器但有大的实现（一个 Postgres 仓库），也可以是大的适配器但有小的实现（一个内存 fake）。当接缝是主题时使用"adapter"；否则使用"implementation"。

**Depth（深度）**
接口处的杠杆——调用者（或测试）每学习一单位接口可以调用的行为量。当一个大量行为位于一个小接口后面时，模块是**深的**。当接口几乎和实现一样复杂时，模块是**浅的**。

**Seam（接缝）** _（来自 Michael Feathers）_
可以在不原地编辑的情况下改变行为的地方。模块接口所在的*位置*。选择在哪里放置接缝本身就是设计决策，与接缝后面的内容分开。
_避免使用_：boundary（与 DDD 的有界上下文重载）。

**Adapter（适配器）**
在接缝处满足接口的具体实现。描述*角色*（它填补什么槽位），而非实质（里面是什么）。

**Leverage（杠杆）**
调用者从深度中获得的东西。每单位他们必须学习的接口有更多能力。一个实现回报于 N 个调用点和 M 个测试。

**Locality（局部性）**
维护者从深度中获得的东西。变更、Bug、知识和验证集中在一个地方，而不是分散在调用者之间。一次修复，处处生效。

## 原则

- **深度是接口的属性，而不是实现的属性。** 一个深度模块可以在内部由小的、可 mock 的、可替换的部分组成——它们只是不是接口的一部分。模块可以有**内部接缝**（对其实现私有，供自身测试使用）以及其接口处的**外部接缝**。
- **删除测试。** 想象删除该模块。如果复杂性消失了，模块没有隐藏任何东西（它是一个透传）。如果复杂性在 N 个调用者处重新出现，该模块确实物有所值。
- **接口就是测试面。** 调用者和测试跨越相同的接缝。如果你想测试*越过*接口，模块的形状可能不对。
- **一个适配器意味着假设的接缝。两个适配器意味着真实的接缝。** 除非有什么东西确实在跨接缝变化，否则不要引入接缝。

## 关系

- 一个 **Module** 恰好有一个 **Interface**（它呈现给调用者和测试的表面）。
- **Depth** 是 **Module** 的一个属性，相对于其 **Interface** 度量。
- **Seam** 是 **Module** 的 **Interface** 所在的位置。
- **Adapter** 位于 **Seam** 处，满足 **Interface**。
- **Depth** 为调用者产生 **Leverage**，为维护者产生 **Locality**。

## 被拒绝的框架

- **深度作为实现行数与接口行数之比**（Ousterhout）：奖励填充实现。我们使用深度作为杠杆替代。
- **"Interface"作为 TypeScript 的 `interface` 关键字或类的 public 方法**：太窄——这里的接口包括调用者必须知道的每个事实。
- **"Boundary"**：与 DDD 的有界上下文重载。说 **seam** 或 **interface**。
