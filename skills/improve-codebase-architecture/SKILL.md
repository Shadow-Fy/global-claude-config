# 改善代码库架构

暴露架构摩擦并提出**深化机会**——将浅层模块重构为深层模块。目标是可测试性和 AI 可导航性。

## 术语表

在每个建议中精确使用这些术语。一致的语言是关键——不要偏离到"component"、"service"、"API"或"boundary"。完整定义见 [LANGUAGE.md](LANGUAGE.md)。

- **Module（模块）**——任何有接口和实现的东西（函数、类、包、切片）。
- **Interface（接口）**——调用者使用该模块需要知道的一切：类型、不变式、错误模式、顺序、配置。不仅仅是类型签名。
- **Implementation（实现）**——内部的代码。
- **Depth（深度）**——接口处的杠杆效应：小接口背后的大量行为。**Deep（深）** = 高杠杆。**Shallow（浅）** = 接口几乎和实现一样复杂。
- **Seam（接缝）**——接口所在的位置；可以在不原地编辑的情况下改变行为的地方（使用此词，而非"boundary"）。
- **Adapter（适配器）**——在接缝处满足接口的具体实现。
- **Leverage（杠杆）**——调用者从深度中获得的东西。
- **Locality（局部性）**——维护者从深度中获得的东西：变更、Bug、知识集中在一处。

关键原则（完整列表见 [LANGUAGE.md](LANGUAGE.md)）：

- **删除测试**：想象删除该模块。如果复杂性消失了，那它只是一个透传。如果复杂性在 N 个调用者处重新出现，那它确实物有所值。
- **接口就是测试面。**
- **一个适配器 = 假设的接缝。两个适配器 = 真实的接缝。**

此技能受项目领域模型的**启发**。领域语言为好的接缝提供了名称；ADR 记录了此技能不应重新争论的决策。

## 流程

### 1. 探索

首先阅读项目的领域术语表和你将要涉及的区域的任何 ADR。

然后使用 Agent 工具配合 `subagent_type=Explore` 来遍历代码库。不要遵循僵化的启发式方法——有机地探索，记录你遇到摩擦的地方：

- 哪些地方理解一个概念需要在许多小模块之间来回跳跃？
- 哪些地方模块是**浅的**——接口几乎和实现一样复杂？
- 哪些地方纯函数只是为了可测试性而被提取出来，但真正的 Bug 隐藏在它们如何被调用中（没有**局部性**）？
- 哪些地方紧密耦合的模块跨越了它们的接缝？
- 代码库的哪些部分未经测试，或难以通过现有接口测试？

对你怀疑是浅层的任何东西应用**删除测试**：删除它会集中复杂性，还是仅仅移动它？"是的，会集中"就是你想要的信号。

### 2. 呈现候选方案

呈现一个编号的深化机会列表。对每个候选：

- **文件**——涉及哪些文件/模块
- **问题**——当前架构为什么导致摩擦
- **解决方案**——纯文字描述会有什么变化
- **收益**——用局部性和杠杆来阐述，以及测试会如何改善

**使用 CONTEXT.md 词汇表描述领域，使用 [LANGUAGE.md](LANGUAGE.md) 词汇表描述架构。** 如果 `CONTEXT.md` 定义了"Order"，就说"Order 接收模块"——而不是"那个 FooBarHandler"，也不是"Order 服务"。

**ADR 冲突**：如果一个候选与现有 ADR 矛盾，只在摩擦真实到值得重新审视该 ADR 时才提出。清晰标记出来（例如 _"与 ADR-0007 矛盾——但值得重新讨论，因为…"_）。不要列出每个 ADR 禁止的理论重构。

不要在此阶段提议接口。问用户："你想探索哪些？"

### 3. 盘问循环

一旦用户选定一个候选，进入盘问对话。与他们一起走完设计树——约束、依赖、深化后模块的形状、接缝后面是什么、哪些测试可以存活。

副作用在决策成型时内联发生：

- **将一个深化的模块命名为 CONTEXT.md 中不存在的概念？** 将术语添加到 `CONTEXT.md`——与 `/grill-with-docs` 相同的规范（见 [CONTEXT-FORMAT.md](../grill-with-docs/CONTEXT-FORMAT.md)）。如果文件不存在，懒创建。
- **在对话中精炼模糊的术语？** 立即更新 `CONTEXT.md`。
- **用户因重要原因拒绝了候选？** 提供一个 ADR，表述为：_"想让我将此记录为 ADR，这样未来的架构评审就不会再次建议它？"_ 仅当该原因未来探索者确实需要了解以避免重复建议时才提供——跳过短暂的原因（"现在不值得"）和自明的原因。见 [ADR-FORMAT.md](../grill-with-docs/ADR-FORMAT.md)。
- **想探索深化模块的替代接口？** 见 [INTERFACE-DESIGN.md](INTERFACE-DESIGN.md)。
